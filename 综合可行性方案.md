## 🔴 P1：数据缺口问题（最关键）

### 1.1 问题分析

```
数据库现状：
├── 总信件数：986封
├── 有全文(full_text)：258封 (26.2%)
├── 无全文：728封 (73.8%)  ← 核心瓶颈
│   ├── 无URL：728封
│   ├── 有URL但无全文：0封
│   └── 全文为空：0封
└── 已翻译：258封 (100% of 有全文的)
```

**根因分析：**
1. 初始爬虫只抓取了258封信的URL和全文
2. 728封信在数据库中只有基础信息（公司名、日期），无URL
3. 无URL → 无法爬取全文 → 无法翻译

### 1.2 解决方案

#### 方案A：从FDA网站补全URL（推荐 ✅）

**原理：** FDA网站有所有3435封警告信的列表，包含URL。我们已有的258封URL可作为匹配模板。

**实施步骤：**

```
Step 1: 抓取FDA网站所有警告信URL
├── 方法：使用浏览器自动化（已验证可行）
├── 工具：Playwright/Selenium
├── 数据：3435条记录
├── 预计时间：30-60分钟
└── 输出：URL列表 (company_name → url)

Step 2: 匹配URL到数据库
├── 匹配字段：公司名称 + 发布日期
├── 模糊匹配：处理名称差异（如"LLC" vs "L.L.C."）
├── 预计匹配率：80-90%
└── 输出：更新warning_letters.url字段

Step 3: 爬取全文
├── 工具：现有fulltext.py爬虫
├── 并发：5线程
├── 延迟：0.5秒/请求（礼貌爬虫）
├── 预计时间：6-10小时（728封 × 0.5秒）
└── 输出：更新warning_letters.full_text字段

Step 4: 重新翻译
├── 工具：batch_translate_v2.py（已优化）
├── 范围：728封新全文
├── 预计时间：2-3天（5线程并发）
└── 输出：更新ai_analysis.translation_zh字段
```

**资源需求：**

| 资源 | 消耗 | 说明 |
|------|------|------|
| API调用 | ~3000次 | NVIDIA NIM翻译调用 |
| 时间 | 3-4天 | URL补全+爬取+翻译 |
| 存储 | <500MB | 临时文件+日志 |
| 人工 | 2-3小时 | 监控+异常处理 |

**风险评估：**

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| FDA网站变更 | 低 | 高 | 定期监控，备用方案 |
| 匹配率低 | 中 | 中 | 手动补全难匹配的 |
| 爬虫被封IP | 低 | 中 | 代理IP池，降低频率 |
| API配额耗尽 | 低 | 中 | 多Key轮询，等待重置 |

**预期结果：**
- 数据覆盖率：26.2% → 95%+
- 翻译覆盖率：26.2% → 95%+

## 🟡 P2：翻译质量问题

### 2.1 问题分析

```
当前质量分布（258封）：
├── A级 (优秀)：182封 (70.5%)
├── B级 (良好)：0封 (0%)
├── C级 (合格)：45封 (17.4%)
├── D级 (较差)：0封 (0%)

```

**F级原因分析：**

| 原因 | 扣分 | 影响 |
|------|------|------|
| AI总结词（"总之"、"综上所述"等） | -25分 | 主要原因 |
| 翻译过短（<200字） | -20分 | 次要原因 |
| 长度比异常（<0.3或>0.9） | -10~30分 | 次要原因 |

### 2.2 解决方案

#### 方案A：优化翻译Prompt（推荐 ✅）

**当前Prompt问题：**
- 虽然明确禁止AI总结词，但模型仍会添加
- 需要更强的约束

**优化措施：**

```python
# 新增到TRANSLATE_PROMPT
【强制检查】翻译完成后，请自查：
1. 是否包含"总之"、"综上所述"、"总结"等词汇？如有，删除
2. 是否添加了原文没有的段落？如有，删除
3. 是否遗漏了原文的任何段落？如有，补充

【输出格式】
直接输出翻译结果，不要包含：
- <thinking>标签
- 任何解释说明
- 任何元数据
```

**实施：** 更新batch_translate_v2.py中的TRANSLATE_PROMPT

**预期效果：** F级占比从12%降至5%以下

## 🟡 P5：长文本截断问题

### 5.1 问题分析

```
截断规则：
├── 阈值：10000字符
├── 处理：截断 + 添加"[...文本过长已截断...]"
├── 影响：长信件翻译不完整
└── 示例：原文19306字 → 翻译2873字 → 评分50(F)
```

**受影响信件：**
- 原文 > 10000字的信件
- 主要是CGMP违规信（内容详细）

### 5.2 解决方案

#### 方案A：分段翻译（推荐 ✅）

**原理：** 将长文本分段翻译，再合并

**步骤：**
```
Step 1: 智能分段
├── 按段落边界分割（\n\n）
├── 每段 < 8000字（留余量）
├── 保留段落标记
└── 输出：段落列表

Step 2: 逐段翻译
├── 每段独立调用API
├── 保持上下文连贯
└── 输出：翻译段落列表

Step 3: 合并结果
├── 按原顺序合并
├── 验证完整性
└── 输出：完整翻译
```

**实施：**
```python
def translate_long_text(text, max_segment=8000):
    """分段翻译长文本"""
    # 按段落分割
    paragraphs = text.split('\n\n')
    segments = []
    current = ''
    
    for para in paragraphs:
        if len(current) + len(para) > max_segment:
            segments.append(current)
            current = para
        else:
            current += '\n\n' + para
    if current:
        segments.append(current)
    
    # 逐段翻译
    translations = []
    for seg in segments:
        result, model = translate_one(seg)
        if result:
            translations.append(result)
    
    # 合并
    return '\n\n'.join(translations)
```

**预期效果：**
- 长信件翻译完整度：60% → 95%+
- F级中因截断导致的：减少50%+
